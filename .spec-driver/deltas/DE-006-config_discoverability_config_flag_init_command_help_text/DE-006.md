## 1. Summary & Context
- **Product Spec(s)**: PROD-001 – interstitial mindfulness logger
- **Technical Spec(s)**: SPEC-002 (`cmd/im`), SPEC-004 (`internal/config`)
- **Implementation Plan**: [IP-006](./IP-006.md)
- **Change Drivers**: Config path is invisible to users — no CLI flag, no init command, no mention in `--help`.

## 2. Motivation
- `im` loads config from `$XDG_CONFIG_HOME/im/im.toml` but never tells the user this.
- No way to override the config path per-invocation.
- No way to bootstrap a config file with defaults — user must know the schema and write TOML by hand.
- NF-001 ("zero config on first run") is satisfied, but FR-003 ("log dir overridable") is hard to exercise without knowing where to put the file.

## 3. Scope & Objectives
- **Primary Outcomes**:
  - `--config PATH` global flag to override config file location (FR-003 enablement)
  - `im init` subcommand writes default `im.toml` to `DefaultConfigPath()` (or `--config` target) with comments documenting each key
  - `--help` output mentions config file location and available config keys
- **Operational Constraints**: Must preserve NF-001 (zero-config still works), NF-002 (< 100ms inline write)

## 4. Out of Scope
- Config file format changes (stays TOML)
- New config keys
- Template init (template.md bootstrapping — separate concern)

## 5. Approach Overview
- **System Touchpoints**: `cmd/im` (CLI flags/subcommands), `internal/config` (init/write support)
- **Key Changes**:
  - `cmd/im`: add `--config` global flag, `init` subcommand, enhance help text (SPEC-002)
  - `internal/config`: add `WriteDefault(path string) error` to write commented defaults (SPEC-004)

## 6. Verification Strategy
- **Requirements Coverage**:
  - `--config` flag: test that custom path is loaded
  - `im init`: test file creation, no-clobber behaviour, content correctness
  - Help text: test output contains config path mention
- **Acceptance Criteria**:
  - `im --help` mentions config location
  - `im init` creates valid, loadable config
  - `im --config /tmp/test.toml init` writes to specified path
  - All existing tests pass; zero lint warnings

## 7. Risks & Mitigations
- **Risk**: `init` clobbers existing config – *Likelihood*: medium – *Impact*: high – *Mitigation*: refuse to overwrite unless `--force` is passed

## 8. Follow-ups & Tracking
- Template bootstrapping (`im init` could also create `template.md`) — defer to future delta
- ISSUE-001: tech specs are stubs; this delta will add substance to SPEC-002/SPEC-004 requirements
