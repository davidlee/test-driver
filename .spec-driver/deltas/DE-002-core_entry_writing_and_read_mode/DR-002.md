# DR-002 – Core entry writing and read mode

## 1. Executive Summary
- **Delta**: [DE-002](./DE-002.md)
- **Status**: draft
- **Last Updated**: 2026-03-04
- **Synopsis**: Implement the core write path (daily file creation, timestamp
  subheadings, entry append) and read mode (`-r`) for `im`, making the tool
  functional for all non-editor input modes.

## 2. Problem & Constraints
- **Current Behaviour**: `im` detects input mode and prints diagnostics but
  writes nothing. The tool is non-functional beyond its scaffold.
- **Drivers / Inputs**: PROD-001 requirements FR-001 through FR-007, FR-009,
  FR-011, FR-012. DE-001 (scaffold) is merged.
- **Constraints / Guardrails**:
  - Append-only — never mutate prior file content (PROD-001 principle).
  - Single-user POSIX tool — no auth, no network, no Windows.
  - < 100ms for inline mode (NF-002).
  - Single statically-linked Go binary (NF-003).
- **Out of Scope**:
  - Editor mode (FR-008, FR-010) — deferred to DE-003.
  - Performance benchmarking — informal verification only.

## 3. Architecture Intent
- **Target Outcomes**:
  - `im hello` creates/appends to `YYYY-MM-DD.md` with date heading + timestamp subheading + entry (FR-001, FR-002, FR-004, FR-005).
  - Configurable log directory with `~/log` default (FR-003).
  - Adaptive and round10 timestamp strategies (FR-004, FR-011).
  - Inline args, `--` separator, and piped stdin all produce entries (FR-005, FR-006, FR-007).
  - `-t` flag produces task checkboxes (FR-009).
  - `-r` displays today's log via viewer chain (FR-012).
- **Guiding Principles**:
  - Three new packages with minimal coupling: `logfile` → `entry`, `reader` standalone.
  - `entry` is a pure function (no I/O, no state). `logfile` owns all file interaction.
  - Testability via dependency injection (clock function), not interfaces.
- **State Transitions / Lifecycle Impact**: No lifecycle states — each invocation
  is stateless. The daily file is the only persistent artifact.

## 7. Design Decisions & Trade-offs

### DEC-001 — File locking: `O_APPEND` only
No `flock`. The kernel guarantees atomic appends for small writes. Single-user
tool — concurrent invocation is unlikely, and the writes are a few hundred
bytes at most. Easy to add `flock` later if needed.

### DEC-002 — Timestamp parsing: read whole file + regex
Daily log files are tiny (1–5 KB). `os.ReadFile` + regex
`^<heading> (\d{2}:\d{2})\s*$` (anchored, strict format). Take the last
match. The heading level marker (default `##`) should be parameterized as a
constant or config option so it can be changed consistently in both the
reader and writer.

### DEC-003 — Timestamp strategy lives in `internal/logfile`
The "should I emit a new heading?" decision requires the last heading time
and the current time — both are `logfile` concerns. Strategy (adaptive vs
round10) is selected from config. Testability via injected clock:
```go
type Appender struct {
    clock func() time.Time
    // ...
}
```
Production passes `time.Now`. Tests pass fixed/stepping clocks.

### DEC-004 — `entry` is pure text formatting
`internal/entry` receives lines and a task flag, returns a formatted string.
It does not know about files, timestamps, or headings. `internal/logfile`
calls `entry.Format` to get the body, then decides what to wrap around it.
Dependency direction: `logfile` → `entry`. `entry` → nothing.

### DEC-005 — Read mode viewer dispatch via `exec`
Dispatch order: `glow --pager` → `rich --markdown --pager` → `$PAGER` → `cat`.
Use `syscall.Exec` to replace the process — the user expects pager behavior,
no reason to keep `im` running. If today's file doesn't exist, print
"no entries for today" and exit 0.

### DEC-006 — Pipe mode: `io.ReadAll`
Single `io.ReadAll(os.Stdin)` call in `cmd/im/main.go`. Streaming is
unnecessary for a logging tool. Trim trailing whitespace, pass to
`entry.Format`.

### DEC-007 — Empty input guard in `run()`
After gathering input (any mode), trim whitespace. If empty, return `nil` —
no file creation, no append, exit 0. Silent. Single gate in `run()`, before
any `logfile` call.

## 9. Rollout & Operational Notes
- **Migration**: None. No prior data to migrate — this is greenfield.
- **Recovery**: Append-only design means no data can be lost by a bad release.
  Worst case: malformed entries that can be hand-edited in the Markdown file.
